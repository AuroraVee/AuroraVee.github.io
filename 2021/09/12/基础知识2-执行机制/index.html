    javascript的执行机制

    1.单线程

    关于JS为什么是单线程，这是有我之前学习时参考的阮一峰老师的博客。

    总的来说，JavaScript作为浏览器脚本语言，它主要用途是与用户互动，以及操作DOM。这决定了它只能是单线程，否则会带来很复杂的同步问题。

    但是为了利用多核CPU的计算能力，HTML5提出Web Worker标准，允许JavaScript脚本创建多个线程，但是子线程完全受主线程控制，且不得操作DOM。所以，这个新标准并没有改变JavaScript单线程的本质。

    2.异步编程

    前面讲到javascript是单线程，需要等上一行代码执行结束才能执行下一行代码（同步）。但是在很多情况下，浏览器是不可能一直等到前面的事情发生才去执行后面的代码，要不然页面会一直卡死。所以javascript可以实现异步，它不会阻塞同步代码的运行。

    JS可以实现的异步编程模式有：

    1. 回调函数：每个任务只能指定一个回调函数
    2. 事件监听：可以绑定多个事件，每个事件可以指定多个回调函数。
    3. 发布订阅：观察者模式
    4. promise对象：每个异步任务返回一个promise对象

    3.执行机制

    「参考资料:微信公众号技术漫谈」

    javascript可以实现异步，会另外开辟一个空间去执行这段异步代码，而这额外的空间是由JS所处的运行环境提供的（浏览器和node）

    3.1 浏览器中的JS

    浏览器都默认提供了一个JavaScript引擎，为JS提供一个运行环境

    内存堆heap，是浏览器为了给代码分配运行内存；图中右侧是调用栈stack，每当运行一段代码JS代码时，都会将代码压入调用栈中，然后在执行完毕以后出栈

    （1）调用栈

    只有同步代码时，浏览器会借助调用栈依次执行

    （2）事件轮询

    当同时存在同步和异步代码时，也就有了多个执行空间，需要依赖事件轮询机制

    浏览器中的各种 Web API 为异步的代码提供了一个单独的运行空间（存放回调函数），当异步的代码运行完毕（比如定时时间到）以后，会将代码中的回调送入到 Task Queue（任务队列）中去，等到调用栈空时，再将队列中的回调函数压入调用栈中执行，等到栈空以及任务队列也为空时，调用栈仍然会不断检测任务队列中是否有代码需要执行，这一过程就是完整的Event Loop 了

    浏览器中常用的宏任务和微任务

    名称 举例（常用）
    宏任务 setTimeout 、setInterval 、UI rendering、script、IO
    微任务 promise 、requestAnimationFrame、MutationObserver回调

    根据相关规定，当调用栈为空时，对于这两个队列的检测情况步骤如下：

    1. 检测微队列是否为空，若不为空，则取出一个微任务入栈执行，然后执行步骤1；若为空，则执行步骤2
    2. 检测宏队列是否为空，若不为空，则取出一个宏任务入栈执行，然后执行步骤1；若为空，直接执行步骤1
    3. ……往复循环

    3.2 node中的JS

    （1）事件轮询

    通过 Node.js 的官方文档可以得知，其事件循环的顺序分为以下六个阶段，每个阶段都会处理专门的任务：

    • timers： 计时器阶段，用于处理setTimeout以及setInterval的回调函数
    • pending callbacks： 用于执行某些系统操作的回调，例如TCP错误
    • idle, prepare： Node内部使用，不用做过多的了解
    • poll： 轮询阶段，执行队列中的 I/O 队列，并检查定时器是否到时
    • check： 执行setImmediate的回调
    • close callbacks： 处理关闭的回调，例如 socket.destroy()、socket.onclose

    以上六个阶段，我们需要重点关注的只有四个，分别是 timers 、poll 、check 、close callbacks。这四个阶段都有各自的宏队列，只有当本阶段的宏队列中的任务处理完以后，才会进入下一个阶段。

    Node.js也有宏任务和微任务之分，

    名称 举例（常用）
    宏任务 setTimeout (timers queue)、setInterval (timers queue) 、setImmediate (check queue)、IO、script
    微任务 Promise 、process.nextTick

    注：process.nextTick优先级大于其它的微任务。事实上Node在遇到这个API时，Event Loop根本就不会继续进行，会马上停下来执行process.nextTick()，这个执行完后才会继续Event Loop。

    ​ nodeV11之后的执行顺序

    （2）setTimeout和setImmediate

    开启事件轮询是需要时间的，并且设置定时器的延时时间可能比实际设置的值大。

    所以setTimeout的回调不一定在setImmediate的回调之前。当开启事件轮询时，定时已经到了，则先执行setTimeout的回调；若开启事件轮询时，定时还没到，timers queue中为空，先执行setImmediate的回调，然后要下一次轮询的时候才能执行setTimeout的回调。

    //setTimeout在setImmediate之前
setTimeout(() => {
    console.log('setTimeout');
}, 0)

setImmediate(() => {
    console.log('setImmediate');
})

let start = Date.now()
// 让同步的代码运行30毫秒，才开启事件轮询
while(Date.now() - start < 30)

    //在第一轮循环到达 poll 阶段时，会将 setTimeout 送入 timers queue，但此时早已跳过了 timers 阶段，所以其只会在下一轮循环时被打印 ；同时 setImmediate 此时被送入了 check queue ，那么在离开 poll 阶段以后就可以顺利得先打印 setImmediate 了
const fs = require('fs');

fs.readFile(__filename, () => {
  setTimeout(() => {
    console.log('setTimeout');
  }, 0);
  setImmediate(() => {
    console.log('setImmediate');
  });
});

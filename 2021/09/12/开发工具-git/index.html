    Git

    参考资料：完全学会GIT GITHUB GIT SERVER的24堂课

    1.版本控制系统VCS

    VCS一开始是采用集中控制和管理的方式控制文档版本，但是这种方法需要锁定文档，等修改完了其他人才能继续修改。

    后来变成分布式控制，大家可以都获取文件下来进行修改，然后送回VCS进行合并，Git就是采用的分布式控制技术

    VCS主要有两套：

    • SubVersion（SVN）
    • Git
      • Git Server主要有Github和Gitlab

    2.Git使用

    1. 新建文件夹，右键打开git bash
    2. 输入 git init，初始化仓库；会生成.git的子文件夹

    git add filename/文件夹名/.：新增文件、把当前文件夹下的所有内容加入Git系统的索引

    git status：查看文件状态

    git commit -m "描述"：提交到本地仓库，或者说是把索引中的文件内容加入文档库

    gitk：打开图形查看模式 gitk --all：可以查看所有分支

    里面有patch和tree两种模式查看不同的commit节点情况

    patch：点击不同的commit节点并且选中想要查看的文件，看到的是当前节点和前一个节点的不同；

    tree：点击不同的commit节点并且选中想要查看的文件，看到的是文件完整的内容

    注：git add 选项的不同含义

    git add .：把新增的和被修改的文件加入Git索引

    git add -u：把被修改和被删除的文件加入Git索引

    git add -A：把新增的、被修改的、被删除的文件加入Git索引

    3.Git的配置文件

    优先级从高到低为：

    1. 文档库.git文件夹下的配置文件，git config user.name/user.email
    2. home directory文件夹下的.gitconfig文件，git config --global user.name/user.email
    3. 安装文件夹下etc/gitconfig配置文件,git config --system user.name/user.eamil

    查看文档库的配置为：git config -l

    删除某个配置文件中的配置项：git config --unset (--global/--system) user.name

    git diff可以查看当前文件夹下的文件和文档库中最新版本的差异

    4.把文件存进文档库

    4.1 Git追踪文件的方式

    • tracked：已经加入文档库
    • ignored：被忽略的，不需要跟踪，需要创建.gitignore文件
    • untracked：没被追踪的，可以通过git status查看没被追踪的文件

    .gitignore文件：

    创建方法touch .gitignore。

    .gitignore文件只负责管理它所在文件夹以及子文件夹下的东西，每个文件夹里都可以创建。

    文件里每一个忽略项各自占一行，可以使用.通配符，#表示注释，!表示不包括

    4.2 控制commit节点

    查看commit提交状态：

    • gitk
    • git log (--graph --oneline)

    每一 个commit节点都有节点标识（很长的16进制数字，一般使用前4位），也可以设置节点标签。其中HEAD标签（简写为@）始终代表最新的一次commit，可以通过git show HEAD查看

    自己设置节点标签：git tag tagName commit的节点标识符/标签；删除节点标签：git tag -d commit的节点标签

    4.3 删除某个已追踪的文件

    1. 先直接从文件夹下删除
    2. 执行git add -A .：会检查文件夹中是否有文件被删除
    3. 执行git commit -m "描述"

    当前commit节点里这个文件被删除，但是历史版本里还是存在

    4.4 从文档库中取文件

    git checkout commit节点标识符/节点标签 文件1 文件2：取出的文件会把当前文件夹里的文件覆盖掉，不会修改HEAD节点

    注：git checkout commit节点：也能让文件夹里的文件恢复到指定状态，但是会进入detachedHEAD状态，此时HEAD会指向该节点。如果此时进行了git add和git commit，会从当前节点长出无名的分支。可以通过把这个无名分支删除或者合并到原来的分支解决这种情况。

    git grep (-i/-c) -e 要查找的字符串1 -e 要查找的字符串2 commit节点：找出包含某一字符串的节点

    4.5 修改文件夹/文件的名字并更新到文档库

    1. git mv 旧名字 新名字
    2. git commit -m "备注"

    4.5 暂存文件夹的文件状态

    当文件改到一半，临时需要做其他事情时，可以把文件夹的内容先暂存起来，等处理结束后再恢复

    1. git stash save：存储当前文件状态与最新commit节点的差异，并把文件状态恢复为最新commit节点的数据
    2. git stash list：可以查看文件暂存状态，以及用来对比的commit节点A、描述内容
    3. 执行其他任务，并把修改的内容提交到文档库
    4. git checkout commit节点A .：从之前的对比commit节点中取出文件
    5. git stash pop：取出暂存的文件并合并到当前文件夹中的文件，操作成功后暂存的文件状态就被清除

    4.6恢复到某个版本

    1. git reset 选项 commit的节点标识符/标签：回到指定的commit节点，不会产生新的commit节点

      --soft：只有文档库中的数据会改变，可以直接执行git commit -m ""

      --mixed：文档库和git索引都会恢复到指定节点状态，默认值，可以直接执行git add 文件

      --hard：文档库、git索引、文件夹中的文件都会恢复到指定状态

      注：恢复之后，commit的历史提交记录都没了

    2. git checkout commit节点 .：不会导致冲突

    3. git revert commit节点：回到指定commit节点的前一个状态，并且会创建新的commit节点。可以在恢复过程中使用git revert --abort取消。这种恢复会导致冲突

    5.建立分支、合并和冲突解决

    5.1 建立分支、查看分支、切换分支

    git branch：查看当前所有的开发分支

    方法一：

    1. git branch 分支名称 commit节点：从指定的commit节点长出分支，如果没写节点则从最新的commit节点长出分支

    2. git checkout 分支名称：切换分支

    方法二：

    git checkout -b 新分支名称 commit节点：创建新分支并自动切换到该分支下

    从当前分支A切换到另一分支B，如果A经过tracked的文件在B中没有，说明这些文件是创建分支B之后加入文档库的，需要将这些文件删除后再切换到B分支；如果A没有tracked而在B中tracked了，则切换到B分支时需要从文档库中取回这些文件

    注：切换分支前，应该把最新的文件内容存入文档库，即执行git commit -m

    5.2 删除分支

    1. 先切换到另一个分支A
    2. git branch -d 分支B：应该先合并分支B，再执行删除
    3. 如果想不合并直接删除，可以执行git branch -D 分支B

    5.3 更改分支的名称

    1. 先切换到该分支
    2. git branch -m 新分支的名称

    5.4 合并分支、解决冲突

    5.4.1 合并分支流程

    1. 假设当前分支为A
    2. 执行git merge B：把分支B合并到分支A，合并后的commit节点是A分支
    3. 合并后，执行git branch -d B将B分支删除

    注：如果合并节点后想反悔，可以执行git reset --hard commit节点进行恢复

    5.4.2 合并分支的两种方式

    1. fast-forward merge：创建新的分支B后，B分支也新的commit，但是原来的分支一直没有，此时只能执行git merge B，合并不会产生新的commit节点。可以通过git merge --no-ff B，修改合并方式为3-way merge
    1. 3-way merge：分支B修改的同时，原来的分支也存在commit，合并之后会产生新的commit节点

    如果没有指定合并方式，Git会自动选择

    5.4.3 冲突解决

    如果两个分支在同一文件的同一位置进行了不同内容的修改，在合并时会产生冲突

    1. 如果执行git merge 分支后，提示合并冲突，可以通过git merge --abort放弃合并
    2. 也可以进入冲突文件中，查找冲突位置进行修改后，执行git add .、git commit -m "描述"，完成合并

    6.使用Rebase指令合并分支，简化commit图（有远程仓库时，不能使用）

    使用多分支进行开发时，为了让其他分支和主分支的代码保持同步，需要定时将主分支的修改合并到当前分支。如果采取前面的merge指令，会使得commit图变得很复杂

    使用Rebase指令，也可以将其他分支的代码合并到当前分支，但是可以简化commit图。

    使用方法和merge一样，git rebase 分支

    在合并完成之前，可以通过git rebase --abort来取消合并，回到合并之前的状态。

    如果已经合并，可以通过git reset --hard commit节点来恢复，但是需要知道合并之前HEAD所在的commit节点。这个可以通过git reflog HEAD/分支名称查看

    7.远程仓库

    之前对文件的修改都是保存在.git形成的本地文档库中（local repository），为了团队开发需要建立远程仓库（remote repository）

    远程Git文档库是一个Bare类型的Git仓库，执行git init --bare Git仓库的文件夹名称就可以创建

    7.1 从远程文档库复制出本地仓库

    git clone 地址

    7.2 本地推送到远程

    git push origin 分支名称A：把指定分支的最新状态推送到远程仓库，不会在配置文件中记录本地仓库分支和远程仓库分支之间的对应关系

    git push -u origin 分支名称或者git push --set-upstream origin 分支名称：会记录

    如果进行了如下配置：

    1. git config --global push.default matching

      执行git push，会把本地文档库中所有曾经传送给远程文档库的分支，都一并更新

    2. git config --all push.default simple

      执行git push，检查当前所在分支是否在配置文件中记录了它对应的远程仓库分支，有则进行更新

    3. git push --all：把本地仓库的所有分支都传送给远程仓库

    如果执行git push时报错，则表明此时远程仓库已被修改，执行git pull，相当于执行git fetch，git merge 当前分支，会**自动拉取远程仓库当前分支的最新状态，并进行合并**。合并可能存在冲突，如果修改后合并成功，再次执行git push就能成功上传。

    8.fork远程仓库

    branch是让程序项目分出另一个版本，fork是复制出另一个Git文档库

    clone复制出的文档库不会保留原来的那个文档库，而fork会记录，并且可以重新合并回去

    git fetch upstream：从fork过来的原始远程仓库获取新的数据

    合并：git checkout 某个分支，git merge upstream/同样的分支
